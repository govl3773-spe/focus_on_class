# 웹캠 기반 실시간 수업 집중도 분류 시스템 최종 보고서

## 1. 프로젝트 개요

본 프로젝트는 웹캠으로 입력되는 사용자의 얼굴 이미지를 바탕으로 수업 집중 상태를 실시간으로 분류하는 시스템을 구현하는 것을 목표로 한다. 분류 대상은 `Attentive(집중)`, `Drowsy(졸림)`, `LookingAway(시선 이탈 / 집중 저하)` 세 가지 상태이다.

화상수업이나 온라인 학습 환경에서는 학생이 실제로 수업에 집중하고 있는지 확인하기 어렵다. 화면에는 사용자가 접속해 있더라도, 졸고 있거나 화면 밖을 보고 있거나 얼굴이 화면에서 벗어나는 경우가 발생할 수 있다. 이 프로젝트는 이러한 상황을 단순 출석 여부가 아니라 얼굴 상태 기반의 집중도 신호로 판단하려는 시도이다.

최종 시스템은 FastAPI 서비스로 구성된다. OpenCV가 웹캠 프레임을 읽고, MediaPipe FaceLandmarker가 얼굴 위치를 검출한 뒤, 얼굴 crop 이미지를 CNN 모델에 입력한다. 이후 단일 프레임 결과에 바로 의존하지 않고 일정 개수의 프레임을 모아 최종 집중 상태를 판단한다.

## 2. 전체 시스템 흐름

프로젝트는 단순히 모델 하나를 학습하는 방식이 아니라, 실시간 서비스에 적합한 데이터와 모델을 단계적으로 만들어 가는 흐름으로 진행되었다.

| 단계 | 핵심 작업 | 목적 |
|---|---|---|
| 데이터 수집 | 웹캠에서 얼굴을 검출하고 crop 이미지 저장 | 모델이 실제 서비스 입력과 비슷한 얼굴 이미지를 학습하도록 구성 |
| 데이터 품질 개선 | quality gate와 상태별 점수 기준 적용 | 클래스 경계가 모호한 이미지를 줄이고 학습 데이터 품질 향상 |
| 데이터셋 분리 | train / validation / test 분리 | 학습, 모델 선택, 최종 평가를 분리해 성능을 객관적으로 확인 |
| 모델 비교 | ResNet, MobileNet, EfficientNet 계열 비교 | 정확도뿐 아니라 속도, 크기, 파라미터 수까지 함께 검토 |
| 최종 후보 재비교 | ResNet18, MobileNetV3 Large, EfficientNet-B0 재평가 | 실시간 서비스 후보 모델을 좁히고 안정성 기준으로 최종 선택 |
| 실시간 서비스 구현 | FastAPI, OpenCV, MediaPipe, PyTorch 연동 | 웹캠 입력부터 최종 상태 판단과 세션 분석까지 연결 |

이 흐름에서 중요한 점은 학습용 데이터, 모델 비교, 실시간 서비스가 서로 따로 놀지 않도록 연결했다는 것이다. 데이터 수집 단계부터 얼굴 crop을 저장했고, 서비스에서도 같은 방식으로 얼굴을 crop해 CNN 모델에 입력하도록 맞췄다.

## 3. 데이터 수집 및 품질 기준

데이터 수집 단계의 핵심은 “많이 찍는 것”보다 “클래스 경계가 선명한 이미지만 저장하는 것”이었다. 웹캠 프레임은 조명, 얼굴 위치, 고개 각도, 눈 감김, 흔들림에 따라 품질 차이가 크기 때문에 단순 캡처만으로는 좋은 학습 데이터가 되기 어렵다.

### MediaPipe FaceLandmarker를 사용한 이유

MediaPipe FaceLandmarker는 얼굴의 존재 여부뿐 아니라 얼굴 랜드마크를 제공한다. 이 프로젝트에서는 얼굴 bbox를 만들고, 눈 뜸 정도, 좌우 회전, 고개 기울기 같은 품질 feature를 계산하기 위해 FaceLandmarker를 사용했다.

단순히 전체 웹캠 이미지를 저장하지 않고 얼굴 bbox 기준으로 crop을 저장한 이유는 모델이 배경이나 옷, 책상 같은 부수적인 정보보다 얼굴 상태에 집중하도록 하기 위해서이다. 최종 서비스에서도 MediaPipe로 얼굴을 찾고 같은 방식으로 crop한 이미지를 CNN에 넣기 때문에, 수집 데이터와 실제 입력 형태를 일치시키는 효과도 있다.

### Quality Gate를 적용한 이유

웹캠 데이터에는 얼굴이 너무 작거나, 흐릿하거나, 너무 어둡거나, 얼굴이 화면 밖으로 벗어난 프레임이 섞일 수 있다. 이런 이미지를 그대로 학습하면 모델은 집중 상태가 아니라 촬영 환경의 노이즈를 학습할 가능성이 높다.

따라서 `1-3_capture2.ipynb`에서는 얼굴 검출 후 다음과 같은 feature를 계산하고, 품질 기준을 통과한 crop만 저장했다.

| Feature | 의미 | 품질 판단에서의 역할 |
|---|---|---|
| `face_area` | 얼굴 bbox가 화면에서 차지하는 비율 | 얼굴이 너무 작게 잡힌 프레임 제외 |
| `eye_open` | 눈 뜸 정도 | 졸림과 집중 상태를 구분하는 핵심 신호 |
| `head_down` | 고개 숙임 정도 | 졸림 또는 집중 저하 신호 |
| `off_center` | 얼굴 중심이 화면 중앙에서 벗어난 정도 | 화면 이탈 또는 자세 불안정 판단 |
| `face_yaw` | 얼굴 좌우 회전 정도 | 옆을 보는 상태 판단 |
| `head_tilt` | 고개 좌우 기울기 | 자세 흐트러짐 또는 집중 저하 판단 |
| `frontal` | 정면성 점수 | 정면 집중 상태 판단 |

### 클래스별 점수 기준

초기에는 `Attentive`와 `LookingAway`가 동시에 높게 나오는 문제가 있었다. 예를 들어 눈은 떠 있지만 얼굴이 조금 돌아간 프레임은 집중처럼 보이기도 하고 시선 이탈처럼 보이기도 한다. 이 문제를 줄이기 위해 수집 로직에서는 집중 저하 증거가 강한 경우 `Attentive` 점수를 억제하는 방식으로 점수식을 조정했다.

핵심 아이디어는 다음과 같다.

| 클래스 | 주요 기준 |
|---|---|
| `Attentive` | 얼굴이 충분히 크고, 중앙에 있으며, 눈이 떠 있고, 정면성이 높고, 고개 숙임과 회전이 작을 때 |
| `Drowsy` | 눈 감김, 고개 숙임, 자세 하강 신호가 강할 때 |
| `LookingAway` | 눈은 떠 있지만 얼굴 회전, 화면 중앙 이탈, 고개 기울기 신호가 강할 때 |

발표 관점에서 이 단계는 “객관적인 얼굴 feature를 기준으로 클래스 경계가 선명한 데이터만 수집한 과정”으로 정리할 수 있다. 즉, 사람이 임의로 이미지를 많이 모은 것이 아니라, 얼굴 위치와 자세 feature를 사용해 학습에 적합한 crop을 선별했다.

## 4. 데이터셋 구성

수집된 crop 이미지는 `dataset_123_split.ipynb`에서 train / validation / test로 분리했다. 원본 `dataset_123` 폴더는 보존하고, 분리 결과는 별도 폴더에 복사하는 방식으로 구성했다.

분할 비율은 train 80%, test 10%, validation 10%이며, `SEED = 42`를 고정해 같은 결과를 재현할 수 있도록 했다.

| 클래스 | Train | Validation | Test | Total |
|---|---:|---:|---:|---:|
| Attentive | 426 | 54 | 53 | 533 |
| Drowsy | 468 | 60 | 58 | 586 |
| LookingAway | 451 | 57 | 56 | 564 |
| 합계 | 1,345 | 171 | 167 | 1,683 |

클래스별 수량은 완전히 동일하지는 않지만 큰 편차 없이 비교적 균형적으로 구성되었다. 이는 모델이 특정 클래스만 과도하게 맞히는 방향으로 학습되는 것을 줄이는 데 도움이 된다.

train / validation / test를 분리한 이유는 각 데이터의 역할이 다르기 때문이다.

| 데이터 | 역할 |
|---|---|
| Train | 모델이 가중치를 학습하는 데이터 |
| Validation | 학습 중 모델 선택과 하이퍼파라미터 판단에 사용하는 데이터 |
| Test | 최종 모델의 일반화 성능을 확인하는 데이터 |

특히 test 데이터는 최종 평가용으로 남겨두어야 한다. 학습과 모델 선택에 계속 사용한 데이터로 최종 성능을 말하면 실제 성능보다 좋게 보일 수 있기 때문이다.

## 5. 모델 비교 실험

모델 비교 단계에서는 CNN 기반 이미지 분류 모델을 여러 계열로 비교했다. 비교 대상에는 ResNet 계열, MobileNet 계열, EfficientNet 계열이 포함되었다.

| 계열 | 비교 이유 |
|---|---|
| ResNet 계열 | 기본적인 CNN 백본으로 성능이 안정적이고 기준 모델로 적합 |
| MobileNet 계열 | 모바일 및 실시간 환경을 고려한 경량 모델 |
| EfficientNet 계열 | 파라미터 효율과 성능의 균형을 고려한 모델 |

`1-7_compare.ipynb`, `1-8_all_model_compare.ipynb`에서는 `resnet18`, `resnet34`, `mobilenet_v3_large`, `efficientnet_b0`, `efficientnet_v2_s` 등을 비교했다. 모든 모델은 같은 데이터셋 구조와 공통 전처리 기준에서 평가되었고, 단순 정확도뿐 아니라 실시간 서비스 적용성까지 고려했다.

비교 지표는 다음과 같다.

| 지표 | 사용 이유 |
|---|---|
| Accuracy | 전체 예측 중 맞힌 비율을 확인 |
| Macro F1-score | 클래스별 성능을 균형 있게 확인 |
| Inference time | 웹캠 실시간 처리 가능성 판단 |
| Model size | 배포 및 로딩 부담 판단 |
| Parameter 수 | 모델 복잡도와 과적합 가능성 판단 |

실시간 웹캠 서비스에서는 단순 정확도만으로 모델을 고르면 안 된다. 사용자는 연속된 프레임에서 결과를 보기 때문에, 모델이 약간 더 높은 점수를 내더라도 프레임별 결과가 흔들리거나 처리 속도가 느리면 서비스 경험은 나빠진다. 따라서 본 프로젝트에서는 성능 지표와 함께 추론 시간, 모델 크기, 파라미터 수를 함께 비교했다.

초기 전체 모델 비교에서는 MobileNetV3 Large가 test 기준으로 가장 좋은 균형을 보였다.

| 관점 | 추천 모델 | 근거 |
|---|---|---|
| 성능 우선 | MobileNetV3 Large | test macro F1 0.8915, accuracy 0.8922 |
| 속도 우선 | ResNet18 | 측정 기준 가장 낮은 ms/image |
| 용량 우선 | EfficientNet-B0 | 파일 크기가 가장 작음 |

이 결과는 “가장 정확한 모델”만 찾는 것이 아니라 “실시간 서비스에 실제로 넣을 수 있는 모델”을 찾는 방향으로 이어졌다.

## 6. 최종 모델 선정

최종 후보 단계에서는 모델 범위를 `ResNet18`, `MobileNetV3 Large`, `EfficientNet-B0` 세 가지로 좁혀 다시 비교했다. 세 모델은 각각 장단점이 뚜렷하다.

| 모델 | 장점 | 단점 |
|---|---|---|
| ResNet18 | 검증 성능이 높고 CNN 기준 모델로 안정적 | 파라미터 수가 많고 실시간 UI에서 예측 흐름이 상대적으로 무거울 수 있음 |
| MobileNetV3 Large | 경량 구조, 빠른 추론, 실시간 입력 변화에 안정적인 편 | 정량 지표에서 ResNet18보다 아주 근소하게 낮을 수 있음 |
| EfficientNet-B0 | 파일 크기와 파라미터 효율이 좋음 | 최종 후보 비교에서 validation 성능이 상대적으로 낮음 |

`2-1_three_model_compare.ipynb`의 최종 후보 재비교 결과는 다음과 같다.

| 모델 | Best Val Accuracy | Best Val Macro F1 | Parameter 수 |
|---|---:|---:|---:|
| ResNet18 | 0.8363 | 0.8286 | 약 11.18M |
| MobileNetV3 Large | 0.8304 | 0.8265 | 약 4.21M |
| EfficientNet-B0 | 0.8070 | 0.8037 | 약 4.01M |

정량 지표만 보면 ResNet18이 가장 높다. 하지만 ResNet18과 MobileNetV3 Large의 validation macro F1 차이는 약 0.0021로 매우 작다. 이 정도 차이는 데이터 분할, 랜덤 시드, 실제 웹캠 입력 조건에 따라 순위가 바뀔 수 있는 수준이다.

따라서 최종 모델 선택에서는 “검증 데이터에서 0.2%p 정도 더 높은가”보다 “실제 웹캠 서비스에서 안정적으로 동작하는가”를 더 중요하게 보았다. 실시간 서비스는 한 장의 이미지를 맞히는 문제가 아니라, 얼굴 검출 → crop → CNN 추론 → 15개 샘플 누적 → 최종 상태 판단으로 이어지는 연속 처리 문제이다. 이 과정에서는 모델의 경량성, 프레임별 확률의 안정성, 서비스 반응성이 중요하다.

이 기준에서 MobileNetV3 Large가 최종 서비스 모델로 가장 적합하다. ResNet18은 정량 성능이 근소하게 높지만, MobileNetV3 Large는 훨씬 작은 파라미터 수로 거의 같은 수준의 성능을 보이며 실시간 UI 적용에 더 유리하다. 즉, 최종 선택은 “점수가 가장 높은 모델”이 아니라 “서비스 환경에서 안정적으로 사용할 수 있는 모델”이라는 기준에서 이루어졌다.

최종 선정 근거는 다음과 같이 정리할 수 있다.

| 기준 | 판단 |
|---|---|
| Validation 성능 | ResNet18이 근소하게 높음 |
| 성능 차이 크기 | ResNet18과 MobileNetV3 Large의 F1 차이가 매우 작음 |
| 모델 경량성 | MobileNetV3 Large가 유리 |
| 실시간 적용성 | MobileNetV3 Large가 유리 |
| 최종 서비스 적합성 | MobileNetV3 Large 선택 |

## 7. 실시간 서비스 구조

최종 서비스는 `service/app.py`의 FastAPI 애플리케이션으로 구현되었다. 사용자가 브라우저에서 서비스를 시작하면 별도 스레드에서 웹캠 프레임을 읽고, 얼굴 검출과 모델 추론을 수행한다.

전체 흐름은 다음과 같다.

```text
웹캠 프레임 입력
→ OpenCV로 프레임 읽기
→ MediaPipe FaceLandmarker로 얼굴 검출
→ 얼굴 bbox 기준 crop 생성
→ PyTorch CNN 모델에 입력
→ Attentive / Drowsy / LookingAway 확률 계산
→ 0.2초마다 샘플 저장
→ 15개 샘플이 모이면 최종 상태 판단
→ SQLite DB에 세션 로그 저장
→ 종료 후 분석 이미지 생성
```

### 15프레임 기반 판단 방식

서비스는 단일 프레임 예측을 바로 최종 결과로 사용하지 않는다. 현재 설정에서는 `SAMPLE_INTERVAL_SEC = 0.2`, `WINDOW_SIZE = 15`이므로 약 3초 단위로 상태를 판단한다.

이 방식을 사용한 이유는 웹캠 프레임 하나에는 흔들림, 눈 깜빡임, 얼굴 검출 실패 같은 순간 노이즈가 많기 때문이다. 한 프레임만 보고 “졸림” 또는 “시선 이탈”로 판단하면 결과가 쉽게 흔들린다. 반면 15개 샘플을 모아 평균과 count를 함께 보면 일시적인 오검출을 줄이고 더 안정적인 상태 판단을 할 수 있다.

### 상태 보정 규칙

서비스의 최종 판단은 단순 평균 argmax가 아니라 보정 규칙을 포함한다.

| 규칙 | 내용 |
|---|---|
| `NoFace` 처리 | 얼굴이 검출되지 않는 샘플이 너무 많으면 `Unknown` 처리 |
| `Uncertain` / `Unknown` 처리 | crop 실패 또는 판단 불가 샘플이 많으면 `Unknown` 처리 |
| `Attentive` 보호 규칙 | 15개 중 일정 수 이상에서 Attentive 확률이 기준 이상이면 최종 상태를 Attentive로 보호 |
| `Drowsy` / `LookingAway` 비교 | Attentive가 충분히 강하지 않을 때 두 비집중 상태의 평균 확률을 비교 |
| `LookingAway` 보너스 | Attentive가 낮고 Drowsy와 LookingAway가 비슷하면 LookingAway 쪽으로 보정 |

이 보정 규칙은 실제 수업 환경에서 상태가 너무 민감하게 바뀌는 문제를 줄이기 위한 장치이다. 특히 눈 깜빡임 하나로 졸림이 나오거나, 얼굴 crop이 잠시 실패해 전체 결과가 흔들리는 상황을 완화한다.

### 세션 로그와 분석 기능

서비스는 최종 판단 결과를 SQLite DB인 `service/attention_logs.db`에 저장한다. 저장되는 정보에는 최종 상태, 클래스별 평균 확률, count, 판단 이유 등이 포함된다.

세션이 종료되면 저장된 로그를 바탕으로 집중 비율, 상태 전환 횟수, 상태별 누적 시간, 구간별 평균 확률을 계산한다. 또한 분석 PNG 이미지를 생성해 사용자가 세션 전체의 집중 흐름을 확인할 수 있도록 했다.

이 구조는 단순히 현재 프레임의 예측값을 보여주는 데서 끝나지 않고, 수업이 끝난 뒤 “언제 집중이 깨졌는지”, “졸림과 시선 이탈이 얼마나 지속됐는지”를 되돌아볼 수 있게 한다.

## 8. 기타 프로젝트 구성과 재현성

프로젝트에는 모델과 실행 환경을 재현하기 위한 파일이 함께 정리되어 있다.

| 구성 | 역할 |
|---|---|
| `models/` | 최종 비교 후 남긴 서비스용 모델 가중치 폴더 |
| `mp_model/` | MediaPipe 얼굴 검출 및 랜드마크 추론용 모델 파일 폴더 |
| `pyproject.toml` | Python 실행 환경과 주요 의존성 정의 |
| `uv.lock` | 의존성 버전을 고정해 같은 환경을 재현하기 위한 lock 파일 |

`models/`에는 ResNet18, MobileNetV3 Large, EfficientNet-B0 가중치가 남아 있어 서비스 적용 전후 비교가 가능하다. `mp_model/face_landmarker.task`는 실시간 얼굴 검출에 필요한 핵심 파일이다.

`pyproject.toml`과 `uv.lock`을 남긴 이유는 재현 가능한 실행 환경을 만들기 위해서이다. CNN 모델, MediaPipe, OpenCV, FastAPI, PyTorch는 버전 차이에 따라 동작이 달라질 수 있으므로, 의존성 파일을 통해 같은 환경에서 다시 실행할 수 있도록 했다.

## 9. 결론 및 향후 개선점

본 프로젝트는 웹캠 기반 수업 집중도 분류 시스템을 데이터 수집부터 실시간 서비스까지 하나의 흐름으로 구현했다. 얼굴 crop 기반 데이터셋을 만들고, quality gate로 클래스 경계가 선명한 샘플을 선별했으며, 여러 CNN 모델을 비교한 뒤 실시간 서비스 적용성을 기준으로 MobileNetV3 Large를 최종 모델로 선정했다.

핵심 결과는 다음과 같다.

| 항목 | 결과 |
|---|---|
| 입력 방식 | 웹캠 기반 얼굴 이미지 |
| 분류 상태 | Attentive, Drowsy, LookingAway |
| 얼굴 검출 | MediaPipe FaceLandmarker |
| 모델 방식 | CNN 전이학습 기반 이미지 분류 |
| 최종 선정 모델 | MobileNetV3 Large |
| 서비스 구조 | FastAPI + OpenCV + MediaPipe + PyTorch |
| 판단 방식 | 15개 샘플 기반 window 판단 |
| 분석 기능 | SQLite 로그 저장 및 세션 분석 이미지 생성 |

다만 현재 시스템에는 개선 여지도 있다. 첫째, 실제 사용자와 조명 환경이 더 다양해지면 모델이 새로운 입력 분포에 흔들릴 수 있다. 둘째, `Drowsy`와 `LookingAway`는 고개 숙임, 회전, 눈 감김이 동시에 나타나는 경우 경계가 겹칠 수 있다. 셋째, 15프레임 window와 threshold 값은 현재 데이터와 테스트 환경에 맞춘 값이므로 더 많은 사용자 환경에서 튜닝할 필요가 있다.

향후 개선 방향은 다음과 같다.

| 개선 방향 | 내용 |
|---|---|
| 추가 데이터 수집 | 다양한 사용자, 조명, 카메라 거리, 자세 데이터를 보강 |
| Threshold 튜닝 | Attentive 보호 기준, LookingAway 보너스, Unknown 처리 기준 재조정 |
| Smoothing window 개선 | 15개 고정 window 대신 상황에 따라 동적으로 조정 |
| 오분류 프레임 재학습 | 실제 서비스에서 틀린 프레임을 수집해 재학습 데이터로 활용 |
| 사용자별 보정 | 얼굴 크기, 카메라 위치, 평소 자세에 따른 개인화 기준 적용 |

최종적으로 이 프로젝트는 단순 이미지 분류 모델 학습을 넘어, 실제 웹캠 입력에서 안정적으로 동작하는 집중도 판단 서비스까지 연결했다는 점에 의미가 있다. 특히 ResNet18이 정량 지표에서 근소하게 높더라도, 실시간 UI에서는 경량성과 안정성이 중요한 기준이 되므로 MobileNetV3 Large를 최종 서비스 모델로 선택한 것이 합리적이다.
